# DEPLOY_CHECKLIST

自分専用本番テスト公開前の確認チェックリスト。

## 1. 現在の前提

- **開発環境**：ローカルMac + MAMP（PostgreSQL Homebrew版）
- **認証**：未実装。固定デモユーザー（`demo@local.hunting-app`）前提
- **ユーザー数**：1名（自分のみ）
- **画像保存**：ローカルファイルシステム（`public/uploads/renewal-permits/`）
- **DB**：ローカルPostgreSQL直接接続
- **UI**：スマホ優先 + PC対応
- **データソース**：手動入力＋スクリプトでダミーデータ作成
- **環境変数**：ローカル .env 設定済み

---

## 2. 自分専用本番テストの目的

実際に運用してみることで：
- 日々の使い勝手を検証する
- データ操作フロー全体を通す（入力→保存→表示→編集→削除）
- 画像アップロード機能を実際に使う
- 本番の推奨スペックや周辺ツール導入を見当つける
- 国営園を想定した環境で一人で回して、画面や操作が実務的かどうか確認する

自分専用本番テスト向けの環境候補：

| 環境 | 特徴 | 適性判定 |
|------|------|--------|

## 5. 本番DB方針

### 5.1 本番 PostgreSQL の選択肢

| 方式 | 特徴 | 自分専用版 | 一般公開版 |
|------|------|---------|---------|
| ローカル PostgreSQL をそのまま | 現状と同じ。セットアップ最小限 | ⭕ 可 | ❌ 不可（バックアップ・アクセス制限弱い） |
| さくら等レンタルサーバーの PostgreSQL | サーバー管理画面から操作 | ⭕ 可 | ⭕ 可（実績ある） |
| Neon（PostgreSQL SaaS） | 無料枠あり。バックアップ自動 | ⭕ 可 | ⭕ 可（スケーラブル） |
| AWS RDS | スケーラビリティ高い。固定費月額あり | ⭕ 可 | ⭕ 可（本番向け） |
| Supabase（Postgres + auth + storage） | 認証・ストレージも込み | ⭕ 可 | ⭕⭕ 強推奨（認証導入時楽） |

**推奨**：本番テスト段階なら可用性低くてもよいので **Neon 無料枠** または **さくら PostgreSQL**。  
認証導入視野なら **Supabase** 早期検討がおすすめ。

## 8. アクセス制限・認証の暫定方針

### 8.1 本番環境でのアクセス制限

| 方式 | 実装 | 自分専用版 | 一般公開版 |
|------|------|---------|---------|
| 固定デモユーザー（現状） | 完了 | ⭕ 可（自分だけ） | ❌ **不可**（認証なし） |
| HTTP Basic認証 | 未実装 | ⭕ 可（簡易） | ❌ **不可**（パスワード無し） |
| Vercel の環境制限（Deployment Protection） | 未実装 | ⭕ 可（本番URLのみ） | ❌ **不可** |
| 本物の認証（GitHub OAuth / NextAuth.js） | 未実装 | ⭕ 可 | ⭕ 可 |
| 固定トークン / API キー認証 | 未実装 | ⭕ 可 | ⚠️ （開発者向け等） |

**自分専用本番テストの方針**：
- ✅ **フェーズ1（今回）**：固定デモユーザーのまま。`getCurrentUser()` が常に demo ユーザーを返す実装を維持。
- ⚠️ **リスク**：本番URL が漏洩したら、誰でもアクセス・データ削除可能。
  - 対策1：ローカル実行のまま、本番化しない
  - 対策2：本番URL を秘密にする（誰にも教えない等）
  - 対策3：HTTP Basic認証等で簡易保護
- ✅ **段階的移行**：テスト期間中は固定デモユーザー、本番公開前に認証導入を視野

**本番テスト期間の推奨設定**：
- ✅ Vercel デプロイなら [Deployment Protection](https://vercel.com/docs/deployments/preview-deployments#deployment-protection) を有効化（パスワード保護）
- ✅ レンタルサーバーなら `.htaccess` + HTTP Basic認証（IP制限等）
- ⚠️ 本番URL を知られないよう、友人への共有は避ける

### 8.2 本番環境での認証設定

- [ ] 本番環境用の認証方針を決定（固定デモユーザー or 簡易保護）
- [ ] アクセス制限が有効になっているか確認（パスワード等）
- [ ] 本番 URL を知人が勝手にアクセスできない工夫

- ❌ **メール送信機能**：初期版では不要

- ❌ **API 公開**：Webアプリのみ
- ❌ **スマートフォンアプリ化**：Webアプリで対応
- ❌ **多言語対応**：日本語のみ
- ❌ **SEO 最適化**：自分専用なのでクローラー不要
- ❌ **キャッシュ CDN**：小規模なため不要（当面）

## 15. 自分専用本番テスト版 vs 一般公開 MVP 比較表

| 項目 | 自分専用テスト版（今回） | 一般公開 MVP（将来） |
|-----|------------------------|--------------------|
| **ユーザー数** | 1（自分のみ） | 多数 |
| **認証** | 固定デモUser | OAuth / 本物の認証 |
| **アクセス制限** | なし（URL知ってるなら誰でも） | パスワード認証必須 |
| **画像保存** | ローカルファイル | S3 / R2 / Supabase |
| **DB** | ローカル or 簡易ホスティング | RDS / Supabase / 専用サーバー |
| **バックアップ** | 手動（月1回） | 自動（毎日） |
| **ログ・監視** | 基本的なサーバーログのみ | Sentry / DataDog 等 |
| **テスト** | 手動確認のみ | 自動テスト + 負荷テスト |
| **ドキュメント** | 開発向けのみ | ユーザーマニュアル必須 |
| **SLA / 稼働率** | 目標なし（落ちてもOK） | 99.9% 以上 |
| **セキュリティ** | 最小限 | 厳格（脆弱性診断等） |
| **スケーラビリティ** | 考慮なし | 複数ユーザー・大容量対応 |
| **サポート** | 自分のメンテナンスのみ | ユーザーサポート体制 |
| **ライセンス / 規約** | 個人的な事項のみ | 利用規約 / プライバシーポリシー公開 |

**重要な判定基準**：

| ルール | 自分専用版 | 一般公開版 |
|--------|---------|---------|
| データ漏洩のリスク | ⚠️ 許容（自分のデータのみ） | ❌ 取得厳禁（他人のデータ） |
| 認証なし運用 | ⭕ 可（自分専用） | ❌ 不可（不正アクセス可能） |
| ローカル画像保存 | ⭕ 可（テスト向け） | ❌ 不可（バックアップ・冗長性不足） |
| バックアップなし | ⭕ 可（やり直せる） | ❌ 不可（ユーザーデータ紛失） |
| SQL 注入等への対策 | ✅ 必須（常に） | ✅ 必須（常に） |
